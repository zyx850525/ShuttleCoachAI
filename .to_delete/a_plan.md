# 🚀 ShuttleCoach AI · Vibe Coding 工作计划

## 🎯 总目标（始终不变）

> 用**最少代码 + 最短路径**
> 跑通：
> **视频 → 动作分析 → 教练反馈 → 可视化结果**

---

## 🧭 Vibe Coding 总原则（先记住这 5 条）

1. **永远只解决“当前这一步”**
2. **永远可运行、可看到结果**
3. **先丑后美，先假后真**
4. **先写死数据，再接模型**
5. **先单动作，再多动作**

---

## 🧱 阶段拆解总览（4 个 Vibe 阶段）

| 阶段      | 目标  | 产出  |
| ------- | --- | --- |
| Phase 0 | 能跑  | 骨架  |
| Phase 1 | 能看  | 关键点 |
| Phase 2 | 能评  | 评分  |
| Phase 3 | 像教练 | 文案  |

---

## 🟣 Phase 0：项目骨架（半天～1 天）

### 🎯 Vibe 目标

> **什么都不智能，但能从头跑到尾**

### 要做什么

* 创建项目目录
* FastAPI 起服务
* 一个 `/analyze` 接口
* 返回假数据

### vibe prompt 示例

> “帮我用 FastAPI 写一个最简单的 `/analyze` 接口，
> 接收视频文件，直接返回一个 mock 的动作分析 JSON。”

### 验证标准

* 能 curl
* 能在浏览器看到 JSON
* 前端能渲染结果

---

## 🟢 Phase 1：关键点可视化（1–2 天）

### 🎯 Vibe 目标

> **“看到动作”比“分析动作”更重要**

### 要做什么

* 用 MediaPipe Pose
* 抽帧
* 画骨架
* 保存一张图

### vibe prompt 示例

> “给我一段 Python 代码，用 MediaPipe 读取视频并画出人体关键点。”

### 验证标准

* 能看到你自己的挥拍骨架
* 你能一眼看出：
  “哦，这一拍是杀球”

---

## 🟡 Phase 2：单动作评分（2–3 天）

### 🎯 Vibe 目标

> **不是准，是“说得通”**

### 只做一件事

👉 **只支持「杀球」**

### 要做什么

* 计算 2–3 个指标：

  * contact_height
  * swing_speed
* 写死阈值
* 给一个分数

### vibe prompt 示例

> “基于 MediaPipe 的关键点，帮我算击球点高度。”

### 验证标准

* 你自己打两次
* 高点 vs 低点
* 分数明显不同

---

## 🔵 Phase 3：教练反馈闭环（1–2 天）

### 🎯 Vibe 目标

> **第一次像教练一样说话**

### 要做什么

* 引入 issue_tag
* 用你设计的 prompt
* LLM 只负责改写

### vibe prompt 示例

> “根据以下结构化信息，用固定模板生成一句教练建议，不要添加新信息。”

### 验证标准

* 你读完会点头
* 没有废话
* 没有跑偏

---

## 🟠 Phase 4：多动作扩展（2–4 天）

### 🎯 Vibe 目标

> **复制，而不是发明**

### 要做什么

* 接入：

  * 高远球
  * 挑球
  * 放网
* 全部走配置
* 不改核心代码

### vibe prompt 示例

> “我已经有一个动作评分引擎，帮我用配置文件扩展到另外 3 个动作。”

### 验证标准

* 新动作 = 新配置
* 不写新逻辑

---

## 🔴 Phase 5：产品感（1–2 天）

### 🎯 Vibe 目标

> **让你愿意给朋友看**

### 要做什么

* 分数卡片
* 问题高亮
* 骨架回放
* 教练总结

### vibe prompt 示例

> “帮我设计一个简单但专业的分析结果页 UI 结构。”

---

## 🧠 Vibe Coding 的节奏建议（非常重要）
